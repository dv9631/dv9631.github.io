what the clauses say:

clause 4: general requiremnts:

  • there must be a QMS system in compliace with ISO 13485
  • there must be a Risk managment system in compliance with ISO 14971
  • software must be classifies into risk class A/B/C
    • CLASS A: no injury or damage to health is possible
    • CLASS B: non-serious injury is possible
    • CLASS C: death or serious injury is possible

Clause 6: Software maintenance process

  1. establish software maintenance plan
    • activities and tasks for software maintenance plan
    • procedures for:
      • recieving,documenting,evaluating resolving and tracking of feedback after release of medical device software
      • criteria for determining feedback is not a problem
      • use of software risk managment process
      • software problem resolution process to analyse and resolve problems after release
      • use of software configuration management process for managing modifications to existing system
      • evaluate and implement upgrades,bugfixes,patches and obsolescence
  2. problems and modification analysis
    • document and evaluate feedback
    • monitor feedback
    • document and evaluate feedback-problem report
    • evaluate problem reports affects in safety
    • use of software resolution process
    • analyse change request for effect on organisation,released software products.and systems with which it interferes
    • approve change request which modify released software products
    • communicate to users and regulators
      • problems in released software products and consequences of continued use
      • nature of available changes to released software products and how to obtain and install
  3. modification implementation
    • use established process to implement modification
    • re release modified software as a full release or a modification kit with changed software

Clause 7: software risk management process

  1. analysis of software contributing to hazardous situvation
    • identify software units that could contribute to a hazardous situvation
    • identify potential causes of contribution to a hazardous situation
      • incorrect or incomplete specfication of funtionality
      • software defects
      • failure or unexpected results from SOUP
      • hardware failures and other software defects that could result in unpredectable software ooperation
      • reasonabily forseable misuse
    • evaluate published SOUP anomaly list
    • document potential causes
    • document sequence of events
  2. risk control measure
    • define risk control measure
    • risk control measures implemented in software
  3. verification of risk control measure
    • verify risk control measure
    • document new sequence of events
    • document traceability
      • from the hazardous situation to software item
      • software item to specific software cause
      • software cause to the risk control measure
      • from risk control measure to verification of risk control measure
  4. risk management of software changes
    • analyse changes to medical device software with respect to safety
      • additional potential causes are introduced
      • additional software risk control measures are required
    • analyse impact of software changes on existing risk control measures
    • perform risk managment activities based on analyses

Clause 9: software problem resolution process

  1. prepare problem reports
    • type 1: corrective, preventive or adaptive to new environment
    • size of change number of device models affected supported accessories affected resources involved time to change
    • Criticality:effect of performance on safety and security
  2. investigate the problem
    • identify cause
    • evaluate relevance to safety
    • create change request to correct the problem or document the rationale for taking no action
  3. Advise relevant parties
  4. use chang control process
  5. maintain records
  6. analyse problems for trends
  7. verify software problem resolution
    • problem resolved and probblem report has been closed
    • adverse trends have been reversed
    • chagne request has been implemented
    • additional problems have been introduced
  8. test documentation contents
